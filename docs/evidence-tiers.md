# Evidence tiers — what gets uploaded, by whom, when

> **Audience.** Operators and compliance owners. Read this once to understand the three evidence tiers, which workflow uploads each, and how the portal's freshness model treats each tier differently.

DevAudit's compliance signal is built on a **three-tier evidence taxonomy**. Each tier is uploaded by a different path, refreshes on a different cadence, and answers a different question for the auditor. Mixing the tiers up — uploading a Tier 1 doc via CI auto-upload, or treating a Tier 3 per-event artefact as project-baseline — is the failure mode this taxonomy exists to prevent.

## The three tiers at a glance

| Tier | What                                    | Upload path                                              | Refresh cadence                                          | Examples                                                                                  | Closes (typical)                                                                                                            |
| ---- | --------------------------------------- | -------------------------------------------------------- | -------------------------------------------------------- | ----------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| **1** | Universal policy that never project-varies | Portal Upload Evidence form `/projects/<slug>/upload` | 365 days; portal renders `Expires YYYY-MM-DD`            | `Test_Policy.md`, `INSTRUCTIONS.md`, `AGENT.md`                                           | Cross-cutting policy clauses (ISO 27001 A.5.1, ISO 29119 §2.5)                                                              |
| **2** | Project-level strategy / SoT documents | Portal Upload Evidence form `/projects/<slug>/upload` | 365 days (180 for AI disclosure)                         | `Test_Strategy.md`, `Test_Architecture.md`, `RoPA.md`, `DPIA.md`, `AI_Disclosure.md`, `Periodic_Security_Review_Schedule.md`, `risk-register.md`, `docs/SRS.md`, `docs/ADR/*.md` | Strategy + governance clauses (GDPR Art. 30 / 35, EU AI Act Art. 13, ISO 27001 A.5.1 / A.8.25, SOC 2 CC2.1)                  |
| **3** | Per-event / per-REQ artefacts          | CI auto-upload                                           | Per release for `*_summary.md`; per event for incidents | `srs-alignment.md`, `architecture-decision.md`, `risk-assessment.md`, `test-execution-summary.md`, `security-summary-<ver>.md`, `RELEASE-TICKET-<ver>.md`, `incident-report-*.md`, `periodic-review.md` | Per-release verification clauses (ISO 29119 §3.5.6 Test completion, SOC 2 CC8.1 Change management, GDPR Art. 33 breach notification) |

## Why the upload paths split

Tier 1/2 docs are **high-judgement content** — your project's actual policy stance, processing activities, architecture choices. The pre-v0.1.39 framework auto-uploaded these as CI evidence and the unedited starter stubs landed in the portal on the first push, flipping policy clauses to COVERED off literal `REPLACE BEFORE …` placeholder text. The portal then misread that as canonical for the first release a real auditor would inspect.

**Resolution (v0.1.39):** Tier 1/2 docs are operator-uploaded via the portal form — every refresh forces a human review of the content before it counts as evidence. The portal renders a `Tier 1 — universal policy` or `Tier 2 — domain strategy` pill on every clause's expectation row so auditors (and operators) see which path a given artefact came from. See [DevAudit-Installer#370](https://github.com/metasession-dev/DevAudit-Installer/issues/370) for the design discussion.

Tier 3 docs are **auto-generated by other workflows** — `incident-report-*.md` by `incident-export.yml` (from labelled GitHub issues), `periodic-review.md` by the quarterly cron, the per-REQ artefacts by the SoT-alignment skills during Stage 3 evidence compilation. CI ownership is appropriate because these aren't authored interactively; the operator reviews them on the portal after the fact, not before they upload.

## Tier 1 — Universal policy

**Authoring.** The framework ships these unchanged in [`sdlc/files/_common/`](../sdlc/files/_common/) — same content in every consumer. `npx @metasession.co/devaudit-cli@latest update` re-syncs them on every framework release. **You do not edit them in your consumer** — if a Tier 1 doc needs project-specific content, that content lives in a sibling Tier 2 doc that points at the Tier 1 baseline.

**Operator action.** Once per project (during onboarding) and once per refresh window (typically annually):

1. Visit `/projects/<slug>/upload` on the portal.
2. Pick `compliance_document` as the evidence type.
3. Pick `_compliance-docs` as the requirement (the baseline namespace).
4. Pick the file from your consumer's repo (e.g. `SDLC/_common/Test_Policy.md`).
5. Optional: leave a one-line description ("annual review — no content changes; same as the framework v0.1.54 baseline").

The portal's framework-coverage matrix has filename-anchored predicates for the four Tier 1 docs — once the row exists with the right filename, the corresponding clauses flip to COVERED.

**Freshness window.** 365 days. The portal renders `Expires 2027-06-11` on the clause's expectation row when within 30 days of expiry it shifts to amber; past the date it becomes red (`expired`). Re-uploading the file from a fresh `update` resets the clock.

## Tier 2 — Project strategy + SoT documents

**Authoring.** Some Tier 2 docs ship as starters (Tier 2 governance starters like `RoPA.md`, `DPIA.md`, `AI_Disclosure.md` via [`bootstrap-governance`](./governance-templates.md)); others are skill-maintained continuously (`docs/SRS.md` by `requirements-aligner`, `docs/ADR/` by `adr-author`, `compliance/risk-register.md` by `risk-register-keeper`); others are project-authored from a starter (`Test_Strategy.md`, `Test_Architecture.md`).

**Operator action.** Same as Tier 1 — operator-uploaded via the portal form at the project-baseline namespace `_compliance-docs`. The framework deliberately splits the upload path here so the operator reviews each refresh before it counts.

**Freshness window.** 365 days for most; 180 days for `AI_Disclosure.md` (the EU AI Act's faster review cadence). The portal's annual review prompt fires 30 days before expiry — your `Periodic_Security_Review_Schedule.md` is the schedule the prompt cross-references.

## Tier 3 — Per-event / per-REQ artefacts

**Authoring.** Auto-generated by other workflows:

- **Per-REQ Tier 3 evidence** (`srs-alignment.md`, `architecture-decision.md`, `risk-assessment.md`, `test-execution-summary.md`) — produced by the SoT-alignment skills during `sdlc-implementer` Stage 3. One file per skill per REQ. Lives at `compliance/evidence/REQ-XXX/`.
- **Per-release Tier 3 evidence** (`security-summary-<version>.md`, `RELEASE-TICKET-<version>.md`) — auto-generated by `generate-security-summary.sh` + `generate-housekeeping-release-ticket.sh` (v0.1.41+) on housekeeping releases; operator-authored for tracked releases (with `sdlc-implementer` Stage 3 writing the stub). Lives at `compliance/pending-releases/`.
- **Per-incident Tier 3 evidence** (`incident-report-<id>.md`) — auto-exported from labelled GitHub issues by `incident-export.yml` (DevAudit-Installer#133). The framework's central stub guard skips unedited templates so the auto-installed `incident-report.md` starter can never flip clauses to COVERED on its own.
- **Per-quarter Tier 3 evidence** (`periodic-review.md`) — auto-generated by the quarterly cron (configured per-project in `sdlc-config.json`).

**Operator action.** Per-REQ artefacts: review the skills' output on the portal after they upload, sign them off via the per-artefact sign-off block. Per-release artefacts: review + sign off the auto-PR opened by `generate-housekeeping-release-ticket.sh` on housekeeping releases (most operator interaction is here). Per-incident: triage the issue to closed-with-`incident` label; `incident-export.yml` does the rest. Per-quarter: review the cron-generated review file before its CI push.

**Freshness window.** Per-REQ artefacts have no time component (they're per-release); per-release / per-incident likewise (the release / incident date is the timestamp). `periodic-review.md` has a 365-day window — its CI cadence is 90 days but the portal accepts up to a year before flipping to expired.

## What `update` does and does not touch

| Tier | `install` first time | `update` refreshes? | Re-syncing overwrites your edits? |
| ---- | -------------------- | ------------------- | --------------------------------- |
| **1** (universal policy)        | Synced from framework                                                          | Yes — every release                                                                                                                       | Yes — these never project-vary                                                                                          |
| **2** (project strategy / SoT)  | Starter dropped via `bootstrap-governance` (opt-in); SoT files maintained by skills | **No** — once you've edited a stub it's yours, the framework does not overwrite                                                            | No                                                                                                                       |
| **3** (per-event / per-REQ)     | Produced on-the-fly by workflows / skills                                       | N/A — Tier 3 is auto-generated content, not framework-owned                                                                                | N/A                                                                                                                      |

The fundamental property: **the operator's edited content is sacred; the framework-owned policy + helper scripts are refreshed**. `update` does not change which files belong to which side — Tier 1 paths are framework-owned, Tier 2/3 paths are operator-owned, period.

## Stub bypass — how the framework prevents starter content from counting as evidence

The framework ships a stub-guard inside [`scripts/upload-evidence.sh`](../scripts/upload-evidence.sh) (DevAudit-Installer#133) — every upload's first step reads the file's banner and aborts with a `SKIPPED unedited starter stub` message if it matches:

```
STARTER TEMPLATE — REPLACE BEFORE …
```

This blocks new uploads of unedited starter content via CI. Pre-guard rows (uploaded before v0.1.36) may still be in the portal's `compliance_evidence` table — the META-COMPLY-side `/api/admin/scrub-stub-evidence` endpoint (devaudit#507) walks the historical rows, reads the first 4 KB of each blob, and flips `metadata.is_unedited_stub = true` on the matches so the matrix calculator excludes them too. Idempotent; superadmin-only; one-shot per portal install.
