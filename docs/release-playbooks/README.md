# Release playbooks

Step-by-step walkthroughs for taking a change all the way to production under the DevAudit SDLC — one playbook per release situation, each written **twice**: once for driving it with an AI agent (Claude Code), and once for doing every step by hand with no AI.

| Playbook | Use it when | Version shape | Stage 1 (Plan) | Independent reviewer | Plan-approval pause |
| --- | --- | --- | --- | --- | --- |
| **[High-risk release](./high-risk-release.md)** | Auth, payments, RBAC, crypto, PII, data handling, audit logging, prod infra — or any of those touched by AI-generated code | `REQ-XXX` (tracked) | Full — plan **required** | **Required** (no self-merge) | **Yes** — before any code |
| **[Low-risk release](./low-risk-release.md)** | Internal tools, UI-only changes, config, no regulated data, no auth changes | `REQ-XXX` (tracked) | Light — plan **skipped** | Not required (self-merge after CI) | No |
| **[Housekeeping release](./housekeeping-release.md)** | `docs` / `chore` / `ci` / `build` / `test` / `compliance` / `revert` — no new requirement | `vYYYY.MM.DD` (housekeeping) | **Skipped entirely** | Not required | No |

These playbooks are the step-by-step companions to [`../change-workflows.md`](../change-workflows.md) (change type → workflow → release shape) and the canonical stage docs that ship into every consumer at [`../../sdlc/files/_common/`](../../sdlc/files/_common/) (`0-project-setup.md` … `5-deploy-main.md`). When a playbook says "run the gates" or "compile evidence", those stage docs are the authoritative detail.

## Pick your playbook in 10 seconds

```
Is there a NEW requirement (a feat / fix / refactor / perf change)?
├── No  → it's docs / chore / ci / build / test / compliance / revert
│        → HOUSEKEEPING release  (vYYYY.MM.DD)
└── Yes → it needs a REQ-XXX (commitlint enforces the tag)
         → How risky is it?
           ├── Auth · payments · RBAC · crypto · PII · data handling · audit
           │   logging · prod infra  (or any of these written/regenerated by AI)
           │        → HIGH-RISK release
           └── Internal tool · UI-only · config · docs-in-code · no regulated data
                    → LOW-RISK release
           (MEDIUM sits between: follow the high-risk playbook but the
            plan-approval pause is optional — see the risk note below.)
```

### Risk tiers (the one rule that drives everything)

| Tier | What it covers | Consequence |
| --- | --- | --- |
| **Low** | Internal tools, no regulated data, no auth changes, UI/config/docs | No implementation plan; **self-merge** after CI passes |
| **Medium** | User-facing features, API changes, PII, new dependencies | Implementation plan required; **independent reviewer** required |
| **High** | Security, payments, RBAC, data handling, authentication, audit logging | Plan **+ plan-approval pause before code**; independent reviewer; deepest evidence |
| **Critical** | High, at maximum ceremony (e.g. AI-regenerated security code) | Same as High; treat as "High with no shortcuts" |

> **The AI bump:** AI involvement raises the risk by one level when the change touches a Medium or High category. AI-generated auth code is therefore **Critical**, and an AI-built user-facing feature is **High**. Source: `sdlc/files/_common/Test_Strategy.md` and `1-plan-requirement.md`.

This repo has three risk-bearing playbooks collapsed into two files: **low-risk** covers Low; **high-risk** covers High/Critical. For a **Medium** change, follow the high-risk playbook — everything applies except the Phase-1 plan-approval pause, which is optional for Medium (Medium passes through automatically unless you force it on).

## The AI prompt vocabulary (used across all three playbooks)

When you drive a release with **Claude Code**, the skills auto-fire from natural language — you type remarkably little. For a whole tracked release you type **two** phrases:

| Moment | What you type (verbatim) |
| --- | --- |
| Start the whole lifecycle (Phases 1→4) | `Implement issue #N under the SDLC.` |
| After UAT is approved on the portal (Phase 5) | `resume REQ-XXX` |
| Confirm the triage classification (when asked) | `Proceed` — or reclassify: `treat this as housekeeping` / `this is HIGH risk` |
| Force the plan-approval pause on any tier | run with `--require-plan-approval` (or `DEVAUDIT_REQUIRE_PLAN_APPROVAL=1`) |

Everything else — `requirements-aligner`, `adr-author`, `risk-register-keeper`, `e2e-test-engineer` — is auto-invoked by the `sdlc-implementer` orchestrator inside those two phrases. You only type a skill's own phrase when doing that slice in isolation:

| Task in isolation | Phrase |
| --- | --- |
| E2E tests | `run the e2e tests and file issues` · `add e2e tests for #N` |
| SRS alignment | `align SRS for REQ-XXX` |
| ADR | `draft an ADR for REQ-XXX` · `does this REQ need an ADR?` |
| Risk-register entry | `open a risk-register entry` · `draft a risk-register entry for REQ-XXX` |
| Governance docs | `create / refresh the RoPA` · `write a DPIA` · `update the AI disclosure` |

**Other agents (Cursor / Windsurf / Gemini CLI / Copilot):** the same workflow runs, but skills don't auto-fire — point the agent at the synced `INSTRUCTIONS.md` and run each stage. The copy-paste per-stage prompts live in [`../../sdlc/files/_common/implementing-an-sdlc-issue.md`](../../sdlc/files/_common/implementing-an-sdlc-issue.md) (§ "Sample prompts"). The **no-AI** track in each playbook is the same stage sequence, run by hand.

## The four-eyes approval lifecycle (every release shape goes through it)

Whichever playbook you follow, the release moves through one state machine on the portal:

```
draft → uat_review → uat_approved → prod_review → prod_approved → released
```

- **CI push** registers the `draft` release and uploads the gate evidence.
- **Submit for UAT review** → `uat_review`.
- **A reviewer approves on the portal** → `uat_approved`. Under the default `dual_actor` mode the approver must be a **different person** than the release creator — this is the four-eyes gate. (`auto_low_risk` mode auto-approves **Low** only; Medium/High always need a human click.)
- The **Release Approval Gate** check on the PR confirms the release is `uat_approved` (or later), which unblocks merge to `main`.
- **Merge to main** deploys to production; `post-deploy-prod.yml` moves the release to `prod_review`.
- **A human approves production** (`prod_approved`) and **marks it Released** — again four-eyes under `dual_actor`.

The mandatory CI gates that every release shape must pass green: **TypeScript 0 errors · SAST (Semgrep) 0 above baseline · dependency audit 0 unaccepted high/critical · E2E all pass · build succeeds.** DevAudit reads pass/fail from the **uploaded evidence**, not the GitHub check — so the CI evidence-upload step is mandatory, not optional.
