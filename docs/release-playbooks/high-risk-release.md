# Playbook: high-risk release

> A **tracked** release (`REQ-XXX`) for a **High** or **Critical** risk change. This is the heaviest path: a plan-approval pause before any code is written, a mandatory independent reviewer, the deepest AI/security evidence, and four-eyes approval at both UAT and production.

| | |
| --- | --- |
| **Use when** | The change touches security, payments, RBAC, crypto, authentication, PII, data handling, audit logging, or production infrastructure — or any of those is written/regenerated by AI (which bumps it to Critical). |
| **Version shape** | `REQ-XXX` (tracked) — a `feat` / `fix` / `refactor` / `perf` commit. commitlint **rejects** these without a `[REQ-XXX]` tag. |
| **Stage 1 (Plan)** | Full. Implementation plan **required**, with a **plan-approval pause before code**. |
| **Reviewer** | A second, independent human reviewer is **required**. The author may **not** self-merge. |
| **Approvals** | Four-eyes at UAT **and** production (`dual_actor`: approver ≠ creator). The AI agent must **not** act as the UAT approver. |
| **Gates** | TypeScript 0 · SAST 0 above baseline · dep-audit 0 unaccepted high/critical · E2E all pass · build — all green. |

There are two ways to run this. **Track A** drives it with Claude Code (you type two prompts). **Track B** is the by-hand, no-AI sequence. Both produce identical evidence and pass through the same gates and approvals. New here? Skim [`README.md`](./README.md) first for the risk tiers and the four-eyes lifecycle.

---

## Track A — with Claude Code (the AI path)

You type **two** prompts for the whole release, plus a plan approval in the middle (the High-risk feature).

### 1. Kick it off

```
Implement issue #N under the SDLC.
```

The `sdlc-implementer` skill triages the issue, classifies risk, and (for High/Critical) **stops after writing the plan**:

- It auto-invokes `requirements-aligner` (SRS-ID alignment), `adr-author` (drafts an ADR if the decision is architectural), and `risk-register-keeper` (opens `RISK-NNN` entries — fires for Medium/High).
- If it asks you to confirm the classification, reply `Proceed` — or correct it: `this is HIGH risk` / `this is CRITICAL`.

### 2. Approve the plan (the plan-approval pause — High/Critical only)

The agent halts and posts the implementation plan, **before any code is written**. Read it — it is far cheaper to change the plan than to refactor shipped code. When you're satisfied, tell it to continue (e.g. `plan approved, continue` / `resume`). To force this pause for any tier in future, run the kick-off with `--require-plan-approval` (or set `DEVAUDIT_REQUIRE_PLAN_APPROVAL=1`).

The agent then runs Phases 2–4 unattended: writes failing tests first, implements against the approved plan, runs all local gates, delegates the E2E pack to `e2e-test-engineer`, compiles the per-REQ evidence (test-execution-summary, **detailed** AI prompt record, security summary, release ticket), pushes to `develop`, and opens the release PR.

### 3. Hard stop at the release PR

Phase 4 ends with the PR open and UAT review requested on the portal. The agent will **not** merge. Its issue comment tells you the next move:

> Implementation complete. PR #M opened. Evidence on portal: <link>. UAT review requested. Resume with `resume REQ-XXX` once UAT approval is granted on the portal.

Now the human work that an AI cannot do for a high-risk change:

- A **second, independent reviewer** reviews the PR + the compliance evidence and approves the release on the portal (`uat_approved`). Under `dual_actor` this person must differ from whoever the release lists as creator. The agent is forbidden from being the UAT approver here.

### 4. Resume for production

Once UAT is approved on the portal:

```
resume REQ-XXX
```

Phase 5 merges the PR to `main`, watches the production deploy, runs the read-only production smoke + security verification, and drives the release to `prod_review`. A human then **Approves Production** and **Marks as Released** on the portal (four-eyes again under `dual_actor`), and the agent closes out the release ticket and the issue.

> **Other agents (Cursor / Windsurf / Gemini / Copilot):** skills don't auto-fire. Point the agent at the synced `INSTRUCTIONS.md` and run the stages one at a time using the copy-paste prompts in [`../../sdlc/files/_common/implementing-an-sdlc-issue.md`](../../sdlc/files/_common/implementing-an-sdlc-issue.md). The sequence — and the human plan-approval, reviewer, and production approvals — are identical to Track B below.

## Track B — by hand (no AI agent)

The same five stages, run manually. Detail for each command lives in `sdlc/files/_common/{1..5}-*.md`; this is the high-risk path through them.

### Stage 1 — Plan the requirement (`1-plan-requirement.md`)

```bash
# 1. The issue + the next REQ id
gh issue view 123
grep -oP 'REQ-\d+' compliance/RTM.md | sort -t- -k2 -n | tail -1

# 2. Add an RTM row (Part B of compliance/RTM.md), risk HIGH
#    | REQ-XXX | #NNN | HIGH | compliance/evidence/REQ-XXX/ | DRAFT | -- | -- |
mkdir -p compliance/evidence/REQ-XXX

# 3. Write the implementation plan — REQUIRED for High/Critical
$EDITOR compliance/evidence/REQ-XXX/implementation-plan.md   # template in 1-plan-requirement.md
```

**⛔ Plan-approval checkpoint (High/Critical).** Do **not** write code until the plan is explicitly approved by a human. This is the manual equivalent of Track A's pause. Then commit the plan:

```bash
git add compliance/evidence/REQ-XXX/implementation-plan.md
git commit -m "chore(compliance): [REQ-XXX] implementation plan

Ref: REQ-XXX"
```

Generate the **HIGH** test-scope (security testing, independent review, pentest consideration, performance, and an AI-involvement block), the test plan, and — if AI generated any code — a detailed AI-use note. Each has a review checkpoint. Then commit:

```bash
git add compliance/RTM.md compliance/evidence/REQ-XXX
git commit -m "compliance: [REQ-XXX] define requirement and test scope - <desc> [RISK: HIGH]

Ref: REQ-XXX
Closes: #NNN"
```

### Stage 2 — Implement & test (`2-implement-and-test.md`)

```bash
git branch --show-current            # must be: develop

# TDD: write failing tests first, then implement against the approved plan.
# Tag the code: @requirement REQ-XXX

# If AI generated code, append a detailed record (High = detailed prompts/outputs):
echo "Prompt summary: ..." >> compliance/evidence/REQ-XXX/ai-prompts.md
echo "Files generated: ..." >> compliance/evidence/REQ-XXX/ai-prompts.md
echo "Date: $(date -I)" >> compliance/evidence/REQ-XXX/ai-prompts.md

# Run ALL local gates — every one must be clean:
npx tsc --noEmit
semgrep scan --config auto [SOURCE_DIR]/ --severity ERROR --severity WARNING
npm audit --audit-level=high
npx playwright test

git add <only the files for this change>
git commit -m "feat: <desc>

Ref: REQ-XXX"
git push origin develop                 # triggers full CI + deploys to UAT
gh run watch                            # confirm CI green
```

### Stage 3 — Compile evidence (`3-compile-evidence.md`)

```bash
gh run list --branch develop --limit 1   # confirm CI green

# Author the per-REQ evidence:
#   - test-execution-summary.md   (ISO 29119-3 Test Completion Report)
#   - security-summary.md         (SAST + dep-audit findings, narrated)
#   - ai-use-note.md / ai-prompts.md  (detailed for High)
# Upload it to the portal:
./scripts/upload-evidence.sh [PROJECT_SLUG] REQ-XXX e2e_result <path>
semgrep scan --config auto [SOURCE_DIR]/ --json > /tmp/sast.json
npm audit --json > /tmp/dep.json
./scripts/upload-evidence.sh [PROJECT_SLUG] REQ-XXX sast_report /tmp/sast.json --git-sha "$(git rev-parse HEAD)"
./scripts/upload-evidence.sh [PROJECT_SLUG] REQ-XXX dependency_audit /tmp/dep.json --git-sha "$(git rev-parse HEAD)"

# RTM → "TESTED - PENDING SIGN-OFF"; write the release ticket:
$EDITOR compliance/pending-releases/RELEASE-TICKET-REQ-XXX.md

git add compliance/RTM.md compliance/pending-releases/RELEASE-TICKET-REQ-XXX.md compliance/evidence/REQ-XXX
git commit -m "compliance: [REQ-XXX] evidence compiled - awaiting review"
git push origin develop
gh run watch --workflow "Compliance Evidence Upload"

# Submit for UAT review (draft → uat_review):
./scripts/submit-for-uat-review.sh [PROJECT_SLUG] REQ-XXX
```

> For a High-risk change whose `risk_class` is in the project's `uat.required_risk_classes`, also do the **UAT-environment verification** (health/smoke/feature checks against the staging URL) and record it in `security-summary.md` before submitting. See `3-compile-evidence.md` Step 10.

### Stage 4 — Submit for review (`4-submit-for-review.md`)

```bash
# Only after: CI green on develop, UAT verified (if applicable), evidence uploaded,
# AND the release is approved on the portal.
gh pr create --base main --head develop --title "feat: <desc>" --body "<SDLC body; link the release ticket>"
gh pr checks <PR>                                   # incl. "Compliance Validation" + "DevAudit Release Approval"
gh pr view <PR> --json mergeable,mergeStateStatus
```

**A second, independent reviewer reviews and approves on the portal** (`uat_review → uat_approved`). The author **must not** self-merge a High-risk change — separation of duties (ISO 27001 A.5.3, SOC 2 CC6.1/CC8.1) requires the second pair of eyes.

### Stage 5 — Deploy to production (`5-deploy-main.md`)

```bash
gh pr merge <PR> --merge --delete-branch=false      # do NOT delete develop
git checkout main && git pull origin main
git checkout develop && git merge main --no-edit && git push origin develop

curl -s [PRODUCTION_URL]/[HEALTH_ENDPOINT]          # read-only prod checks only
# Run the release ticket's Post-Deploy Actions; prod smoke + security verification
# (401/403 on protected routes, security headers, no stack traces) → record in security-summary.md
```

The deploy moves the release to `prod_review`. A human **Approves Production** (`prod_approved`) and **Marks as Released** on the portal — four-eyes again. Then close it out:

```bash
./scripts/close-out-release.sh REQ-XXX --release-pr <release-PR-#>   # refuses unless the portal says "released"
git add -A && git commit -m "docs(compliance): close out REQ-XXX release ticket (RELEASED)" && git push origin develop
gh issue close <ISSUE> --comment "Shipped in REQ-XXX. Released and verified in production."
git checkout main && git merge develop --no-edit && git push origin main
```

## What makes this the *high-risk* path (vs low-risk)

- **Plan-approval pause before code.** High/Critical halts after the implementation plan for explicit human sign-off. Low and Medium pass straight through.
- **Independent reviewer, no self-merge.** A second human must approve. Low-risk changes self-merge on green CI.
- **Deepest evidence.** HIGH test-scope adds security testing, independent review, pentest consideration, and performance; the AI record is detailed prompts/outputs (Low is just a `Co-Authored-By` tag).
- **`dual_actor` enforced at both gates.** The UAT approver and the production approver must each differ from the creator; the AI agent is never the approver.

## Quick reference

| Step | Track A (Claude) | Track B (manual) |
| --- | --- | --- |
| Start | `Implement issue #N under the SDLC.` | Stage 1: plan + HIGH test-scope, commit |
| **Plan approval** | review plan → `resume` | **⛔ human approves plan before code** |
| Build + test + evidence | (automatic) | Stages 2–3: TDD, gates, upload, release ticket |
| Submit for UAT | (automatic, hard stop) | `submit-for-uat-review.sh`, open PR |
| **UAT approval** | independent reviewer approves on portal | independent reviewer approves on portal |
| Ship | `resume REQ-XXX` | Stage 5: merge, prod checks |
| **Prod approval** | human Approves Production + Marks Released | human Approves Production + Marks Released |
| Close out | (automatic) | `close-out-release.sh`, close issue |

## Hotfix scenario

A critical production defect needs an emergency fix. The normal develop → UAT → main cycle is too slow. For a high-risk hotfix, the plan-approval pause still applies but can be compressed — a chat message or inline plan in the PR description counts as approval. **Gates are never skipped.**

### Track A — AI-driven hotfix

```text
Hotfix issue #N — production is broken. Implement under the SDLC with --require-plan-approval.
```

The agent will:

1. Branch from `main` (not `develop`) — `fix/hotfix-<desc>`
2. Classify as HIGH (most hotfixes are — they touch prod)
3. Write a **minimal** implementation plan (inline, not the full template)
4. **Pause for plan approval** — you approve in chat (`plan approved, continue`)
5. Fix + write tests + run ALL gates locally
6. Push and open a PR to `main` (or to `develop` then cherry-pick)
7. Compile brief but complete evidence: `test-execution-summary.md` + `security-summary.md`
8. Submit for UAT review on the portal

### Track B — manual hotfix

```bash
# 1. Branch from main
git checkout main && git pull origin main
git checkout -b fix/hotfix-<desc>

# 2. Minimal plan (inline in the PR description is acceptable for a hotfix)
#    Write it BEFORE coding — even a 3-line plan counts.
#    Plan-approval: a chat message from a second human counts as approval.

# 3. Fix + tests + ALL gates (no shortcuts)
npx tsc --noEmit
semgrep scan --config auto [SOURCE_DIR]/ --severity ERROR --severity WARNING
npm audit --audit-level=high
npx playwright test

# 4. Commit with REQ tag (you still need a REQ — file an issue first if none exists)
git add <files>
git commit -m "fix: <hotfix description>

Ref: REQ-XXX
Co-Authored-By: <if AI>"

# 5. PR — can target main directly for a hotfix, or develop then cherry-pick
gh pr create --base main --head fix/hotfix-<desc> \
  --title "fix: <hotfix description>" \
  --body "HOTFIX — production critical. Plan approved inline. Ref: REQ-XXX"

# 6. Evidence (brief but must exist)
#    test-execution-summary.md — which tests ran, pass/fail
#    security-summary.md — SAST clean, dep-audit clean, what the fix changes

# 7. Submit for UAT, approve, merge, prod checks
./scripts/submit-for-uat-review.sh [PROJECT_SLUG] REQ-XXX
# Reviewer approves on portal → merge → prod deploy → prod checks

# 8. Sync develop
git checkout develop && git merge main --no-edit && git push origin develop
```

### Post-hotfix actions

- File a follow-up issue for any root-cause fix that's too large for the hotfix
- Update the risk register if the hotfix reveals a new risk
- If the hotfix was for a regression from a recent release, supersede that release on the portal (see below)

## Superseding a previous release

A release was merged to `develop` or `main` but has a defect. A new release replaces it.

### When the superseded release is still on `develop` (not merged to `main`)

1. **Fix-forward on `develop`:** add a new commit fixing the defect, same REQ
2. The existing release PR picks up the fix automatically — no new PR needed
3. No portal action — the release record stays in its current state
4. Re-run CI to confirm gates still pass

```bash
# On develop, same REQ
git add <fix files>
git commit -m "fix: <defect description> — supersedes broken commit <sha>

Ref: REQ-XXX"
git push origin develop
gh run watch
```

### When the superseded release is merged to `main` but not yet `released` on the portal

**Option A — revert and re-release (preferred for HIGH risk):**

```bash
# 1. Revert the merge commit on main
git checkout main && git pull origin main
git revert -m 1 <merge-commit-sha>
git push origin main

# 2. Fix on develop
git checkout develop
# ... fix the defect ...
git commit -m "fix: <defect description> — fixes regression from REQ-XXX

Ref: REQ-XXX"
git push origin develop

# 3. New release PR
gh pr create --base main --head develop --title "fix: <desc> (supersedes REQ-XXX)" --body "..."
```

**Option B — fix-forward (for defects that don't warrant a revert):**

```bash
# Fix on develop, new release PR with the fix included
git checkout develop
# ... fix ...
git commit -m "fix: <defect description>

Ref: REQ-XXX"
git push origin develop
gh pr create --base main --head develop --title "fix: <desc>" --body "Supersedes the broken merge <sha>. Ref: REQ-XXX"
```

1. Mark the old release as **SUPERSEDED** on the portal (if the portal supports it)
2. The new release carries its own full evidence — do not reuse the old release's evidence

### When the superseded release is already `released` on the portal

This is a **rollback**, not a supersede. See the rollback scenario below.

## Triage: defect found during feature → develop

You're implementing a feature on `develop` (Stage 2) and a defect is discovered — either in the new code or in existing code that the feature touches.

### Case 1: Defect is in the new code being written (same REQ)

1. **Fix in-place** on the same branch, same REQ
2. Add or update tests to cover the defect
3. Re-run all gates
4. Continue with the normal flow — no new REQ, no new PR

```bash
# On develop, same REQ
git add <fix files>
git commit -m "fix: <defect description> in <feature area>

Ref: REQ-XXX"
```

### Case 2: Defect is in existing code, touched by the feature but not caused by it

1. **File a new issue** for the pre-existing defect
2. Decide: fix it now (as part of this REQ) or defer to a new REQ
   - **Fix now:** if the defect blocks the feature or is small enough to fix inline
   - **Defer:** if the fix is large, unrelated, or would expand the scope/risk of the current REQ
3. If fixing now, add a note to `implementation-plan.md` documenting the scope expansion
4. If deferring, file the issue and continue with the feature

```bash
# File the issue
gh issue create --title "Defect: <description>" --body "Found during REQ-XXX implementation. Pre-existing, not caused by the feature."

# If fixing now:
git commit -m "fix: <defect description> — pre-existing, found during REQ-XXX

Ref: REQ-XXX"
```

### Case 3: Defect is in a different area (found by coincidence, e.g. E2E regression)

1. **File a new issue** with the E2E regression label
2. **Do not block the current feature** unless the defect is in a shared component
3. If the defect is in a shared component that the feature depends on:
   - Fix it as part of the current REQ (document the scope expansion)
   - Or block the feature PR until a separate fix REQ is merged
4. If the defect is unrelated, continue with the feature and handle the regression separately

```bash
# File the regression issue
gh issue create --title "E2E regression: <spec name> — <error>" --label "bug" --body "..."
```

## Triage: defect found during develop → main (release PR)

The release PR (`develop → main`) is open and CI fails, or a reviewer finds a defect.

### Case 1: CI gate failure (TypeScript, SAST, dep-audit, E2E, build)

1. **Identify the failing gate** from the CI run
2. **Fix-forward on `develop`:** push a fix commit to `develop`
3. The release PR automatically picks up the fix
4. Re-run CI — all gates must pass before merge

```bash
# On develop
# ... fix the failing gate ...
git add <fix files>
git commit -m "fix: <gate failure description>

Ref: REQ-XXX"
git push origin develop
gh run watch
```

### Case 2: Compliance validation failure (commit convention, missing evidence)

1. **Identify the validation error** from the CI log
2. Common fixes:
   - Missing `Ref: REQ-XXX` on a `fix`/`feat` commit → add a retroactive compliance commit with the REQ reference and a release ticket in `compliance/pending-releases/`
   - Missing evidence artifact → create the artifact and re-run `compliance-evidence.yml`
   - `REPLACE` placeholders in release ticket / security summary → fill them in and push
3. Push the fix to `develop` — the release PR picks it up

```bash
# Example: retroactive REQ attribution for a fix commit that lacked Ref:
git add compliance/RTM.md compliance/evidence/REQ-XXX/ compliance/pending-releases/RELEASE-TICKET-REQ-XXX.md
git commit -m "compliance: [REQ-XXX] retroactive attribution for fix commit <sha>

Ref: REQ-XXX"
git push origin develop
```

### Case 3: E2E regression found on the release PR

1. **Triage the failure:** is it caused by this release's changes or a pre-existing flaky test?
2. **If caused by this release:** fix-forward on `develop` (same REQ)
3. **If pre-existing flaky test:** file a regression issue, decide whether to block the PR
   - **Block:** if the regression is in a critical path that this release touches
   - **Don't block:** if the regression is in an unrelated area — file the issue and merge the PR
4. **If the E2E test is flaky (timing issue):** apply the same fix pattern as REQ-092 — replace `waitForTimeout` with proper `toBeVisible` assertions with adequate timeouts

```bash
# Fix-forward for a regression caused by this release
git checkout develop
# ... fix the E2E test or the underlying code ...
git commit -m "test: fix E2E regression in <spec name> — <root cause>

Ref: REQ-XXX"
git push origin develop
gh run watch
```

### Case 4: Reviewer finds a defect during manual review

1. **Assess severity:** is it a blocker or a suggestion?
2. **Blocker:** fix-forward on `develop`, re-request review after CI passes
3. **Non-blocker:** file a follow-up issue, merge the PR, fix in the next release
4. **If the fix changes the risk profile** (e.g. a HIGH-risk change becomes more complex):
   - Update `implementation-plan.md` with the scope change
   - Re-run the plan-approval pause if the change is significant

## Rollback scenario

The release was merged to `main`, deployed to production, and a critical bug was discovered. The release needs to be rolled back.

### Steps

```bash
# 1. Revert the merge commit on main
git checkout main && git pull origin main
git revert -m 1 <merge-commit-sha>
git commit -m "revert: rollback REQ-XXX — critical production bug

Ref: REQ-XXX"
git push origin main

# 2. Production auto-deploys the revert; verify the fix
curl -s [PRODUCTION_URL]/[HEALTH_ENDPOINT]

# 3. Sync develop
git checkout develop && git merge main --no-edit && git push origin develop

# 4. Mark the release as WITHDRAWN on the portal (if supported)
#    Or add a note to the release ticket: "ROLLED BACK — see revert commit <sha>"

# 5. File a follow-up issue for the proper fix
gh issue create --title "Fix: <root cause of the rolled-back bug>" --body "Rolled back REQ-XXX in commit <sha>. Root cause: ..."

# 6. Close the original issue if it was closed by the release
gh issue reopen <ORIGINAL_ISSUE> --comment "Rolled back due to critical bug. Reopening for fix."
```

### Post-rollback

- The revert commit is a `revert` type — it's exempt from the `[REQ-XXX]` tag, but you should include `Ref: REQ-XXX` for traceability
- The follow-up fix gets a new REQ (or reuses the original if the issue was reopened)
- Update the risk register if the rollback reveals a new risk
- Conduct a post-mortem if the bug reached production — document in an incident report
