# Compliance gates — the five CI workflows

> **Audience.** Operators and engineering leads. Read this once to know what each CI workflow does, when it fires, what it gates on, and how to interpret a red badge.

The DevAudit framework's second pillar is a set of GitHub Actions workflows that ship with the SDLC and get rendered into your project on `npx @metasession.co/devaudit-cli@latest install`. They enforce the lifecycle — independent of the developer's local hooks, independent of the AI agent — so the portal sees the same evidence shape regardless of who or what produced the change.

There are **five workflows**. The first four run in the consumer's own CI and upload evidence to the portal; the fifth runs after merge to main and finalises the release.

## At a glance

| Workflow                       | Trigger                                                 | Owns                                                                                                                                                                                  | Required check on the release PR?       |
| ------------------------------ | ------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------- |
| `ci.yml`                       | Push to `develop`                                       | Quality gates: TypeScript / lint / SAST / dependency-audit / unit / e2e / build. Registers the release on the portal. Uploads gate evidence + per-AC `evidenceShot` screenshots.    | **Yes** — required: `Quality Gates`     |
| `compliance-evidence.yml`      | Push to `develop` (touching `compliance/`) + `workflow_run` after `E2E Regression` | Uploads committed compliance docs + per-REQ Tier 3 evidence (impl plan, test-execution-summary, srs-alignment, architecture-decision, risk-assessment); uploads E2E Regression's results to the portal. | **Yes** — required: `Upload Compliance Evidence` |
| `compliance-validation.yml`    | Pull request to `main`                                  | Validates artefact presence (RTM, implementation plan, evidence pack) + commit-message conventions (`Ref: REQ-XXX`, `Co-Authored-By`).                                                | **Yes** — required: `Compliance Validation` |
| `check-release-approval.yml`   | Pull request to `main`                                  | Blocks the merge until the release status on the portal is `uat_approved` (or further along the lifecycle).                                                                          | **Yes** — required: `DevAudit Release Approval` |
| `post-deploy-prod.yml`         | Push to `main` (post-merge)                             | Runs production smoke + uploads `environment=production` evidence + advances each in-scope release to `prod_review` so a reviewer can mark it Released.                              | No — fires after merge                  |

A sixth, project-owned workflow lives alongside these but is **not** generated by DevAudit-Installer — `e2e-regression.yml` (3-tier E2E gating, v0.1.53+). See [`e2e-test-tiers.md`](./e2e-test-tiers.md) for that one.

## `ci.yml` — Quality Gates (the workhorse)

**Fires on:** `push: branches: [develop]` (matches your project's integration branch from `sdlc-config.json`).

**What it does, in order:**

1. **TypeScript** — `tsc --noEmit` (Node stacks) / `mypy` (Python stacks). 0 errors required.
2. **SAST** — `semgrep scan --config auto`. Failure if any high/critical finding.
3. **Dependency audit** — `npm audit --audit-level=high` (Node) / `pip-audit` (Python).
4. **Unit + integration tests** — `npx vitest run` or `pytest`.
5. **E2E gate (smoke tier)** — `npx playwright test --project=smoke`. The full critical + regression tiers run elsewhere (see [`e2e-test-tiers.md`](./e2e-test-tiers.md)).
6. **Build** — `npm run build` or stack-equivalent.
7. **Register the release** — calls the portal's `/api/ci/releases` endpoint to record a release record keyed by the version derived from the latest commit ([`derive-release-version.sh`](../sdlc/files/_common/scripts/derive-release-version.sh) — `[REQ-XXX]` → tracked, otherwise `vYYYY.MM.DD` housekeeping).
8. **Upload evidence** — each gate's machine-readable output is posted via [`scripts/upload-evidence.sh`](../scripts/upload-evidence.sh) (`sast-results.json`, `dependency-audit.json`, `e2e-results.json`, `playwright-report/index.html`, `coverage-summary.json`).
9. **Upload per-AC screenshots** — for each in-scope REQ (one per file in `compliance/pending-releases/RELEASE-TICKET-REQ-*.md`), uploads the `evidenceShot(page, reqId, ac, slug)` PNGs from `compliance/evidence/REQ-XXX/screenshots/`. Per-REQ scoped — out-of-scope legacy folders are intentionally ignored (DevAudit-Installer#147). Filename validation: `REQ-XXX-AC<n>-<slug>.png`. Failures bump `UPLOAD_FAILURES` and turn the step RED — silent screenshot drops are the failure mode this catches.
10. **CI Status Fallback** — a sibling workflow (`ci-status-fallback.yml`) that flags the consumer's branch protection as green if and only if every required workflow ran (catches the GHA edge case where a skipped workflow leaves a required check in "pending" forever).

**What turns it red:** any gate failure, any non-bypassable lint failure, any upload-evidence.sh non-zero exit. Pre-existing `// @ts-expect-error` and `eslint-disable` lines do **not** silently pass — the framework's hooks block them at commit time (`SDLC rule: NEVER use --no-verify`).

**What turns it amber:** none. Gates are binary — green or red. The portal's "amber within 30 days" is purely about freshness, not CI outcome.

## `compliance-evidence.yml` — the upload pipeline

**Fires on:**
- `push: branches: [develop]` with `paths: ['compliance/**']` — touching the docs.
- `workflow_dispatch` — manual re-upload.
- `workflow_run: workflows: ['E2E Regression']` (DevAudit-Installer#149) — listens for the consumer's project-owned `e2e-regression.yml` completing on PR-to-main or push-to-main, downloads its artefacts, and uploads them to the portal so the UAT four-eyes reviewer sees the critical-tier + full-regression results in the same place as the smoke-tier ones.

**Two jobs inside it:**

- **`upload-compliance-evidence`** — fires on the `push` / `workflow_dispatch` paths. Walks `compliance/governance/` (Tier 3 per-event docs: `incident-report*.md` → `incident_report`, `periodic-review.md` → `periodic_review`) + `compliance/evidence/REQ-*/` (Tier 3 per-REQ artefacts: `srs-alignment.md` → `srs_alignment`, `architecture-decision.md` → `architecture_decision`, `risk-assessment.md` → `risk_assessment`, `test-execution-summary.md` → `test_report`, everything else → `compliance_document`). DevAudit-Installer#146 routes the SoT-alignment artefacts to their dedicated types so the framework-coverage matrix attributes them correctly (without that fix, SOC2.CC3.2 and ISO27001.A.8.25 misreported despite the files being present).

- **`upload-e2e-regression-evidence`** — fires on the `workflow_run` path only. Downloads the prior `E2E Regression` workflow's `e2e-regression-report` artefact via `actions/download-artifact@v4` `run-id`, derives the release version against the triggering SHA, and uploads `e2e-regression-results.json` (as `e2e_result`) + `playwright-report/index.html` (as `test_report`) against each in-scope REQ. Tier metadata (`critical` for the PR-to-main run, `regression` for post-merge) ships in `--meta-key tier=…` so the portal can group by tier on the release detail.

**What is NOT uploaded by this workflow:** the project-baseline Tier 1/2 governance docs (RoPA, DPIA, AI disclosure, the Test_Policy / Test_Strategy / Test_Architecture / Periodic_Security_Review_Schedule). Those are operator-uploaded via the portal's `/projects/<slug>/upload` form — auto-uploading them as CI evidence (the pre-v0.1.39 behaviour) let unedited starter stubs flip clauses to COVERED on day one. See [`evidence-tiers.md`](./evidence-tiers.md) for the full upload-path split.

## `compliance-validation.yml` — the PR-time artefact check

**Fires on:** `pull_request:` against `main`.

**What it checks:**

- RTM row exists for every `REQ-XXX` mentioned in the PR's commits.
- An implementation plan exists at `compliance/plans/REQ-XXX/implementation-plan.md` for every MEDIUM/HIGH-risk tracked REQ.
- An evidence pack exists at `compliance/evidence/REQ-XXX/` with at least the framework-attribution Tier 3 artefacts.
- Every `feat` / `fix` / `refactor` / `perf` commit carries `[REQ-XXX]` in the subject or `Ref: REQ-XXX` in the body (`validate-commits.sh`).
- Every commit's `Co-Authored-By:` trailer is present where the change was AI-assisted.

**What turns it red:** an in-scope REQ with no plan, an untagged behavioural commit, a missing Tier 3 artefact for a tracked REQ. **Housekeeping commits are exempt** — `chore` / `ci` / `build` / `test` / `compliance` / `revert` don't carry a REQ.

**What it does NOT check:** content quality. It verifies presence + naming, not whether the impl plan actually addresses the AC. That's the human reviewer's job at the four-eyes UAT step.

## `check-release-approval.yml` — the merge blocker

**Fires on:** `pull_request:` against `main`.

**What it does:** derives the release version from the PR's commits ([`derive-release-version.sh`](../sdlc/files/_common/scripts/derive-release-version.sh) — same helper as `ci.yml`), then polls the portal for that release's status. Passes when the status is `uat_approved` / `prod_review` / `prod_approved` / `released`. Fails on `draft` / `uat_review` / `change_requested` with the explicit message _"Release REQ-XXX is 'draft' — release approval required (have an authorised reviewer approve in DevAudit)"_.

**Why this gate is necessary:** it's the four-eyes regulatory control made structural. The PR's other checks can be all green and the merge will still be blocked until a human (or a second AI actor for solo teams — see the `dual_actor` rule on the portal's [Releases & approvals](https://github.com/metasession-dev/devaudit/blob/main/docs/releases-and-approvals.md) doc) has clicked Approve. Operators on a solo team must document the `solo_with_gap` interpretation on the release ticket; the `risk-register-keeper` skill enforces the control-gap entry exists.

**Recovery when red:** open the portal at `/projects/<slug>/releases/<id>`, review the evidence, click Approve UAT (or Reject + open change-request loop). The CI check re-evaluates on the next push to the PR.

## `post-deploy-prod.yml` — production verification

**Fires on:** `push: branches: [main]` — i.e. after the release PR merges.

**What it does:**

1. **Wait for the host's deploy** — `wait-for-deploy.sh` polls the host's API until the production deployment of the merge commit is ready (Railway / Vercel / Render — selected by your project's `sdlc-config.json:host`).
2. **Production smoke tests** — runs `e2e/smoke/` against the production URL using credentials from the host's environment + the `production` Playwright project.
3. **Upload production evidence** — every screenshot + test report from the smoke pack is uploaded with `--environment production` so the portal can distinguish UAT-environment evidence (committed during Stages 1–4) from production-environment evidence (this workflow).
4. **Advance the release** — for every in-scope REQ in the merge commit (a bundled deploy can carry several), calls the portal to flip the release status from `uat_approved` → `prod_review`. The operator then clicks Approve Production to mark it `released`.

**What turns it red:** smoke failure against production. The release stays at `uat_approved` (not `prod_review`) and the operator has the decision of whether to roll back, hotfix forward, or mark as accepted-with-rationale on the release ticket.

## How they interact during a tracked release

```
day 1  developer commits + pushes to develop
       └── ci.yml runs → registers REQ-XXX release → uploads gate evidence

day 1  developer commits compliance docs (impl plan, evidence pack, release ticket)
       └── compliance-evidence.yml runs → uploads Tier 3 evidence

day 2  developer opens PR develop → main (the release PR)
       ├── compliance-validation.yml runs → checks artefact presence
       └── check-release-approval.yml runs → polls portal for status (FAIL while draft)

day 2  developer optionally pushes e2e-regression run on the PR
       └── E2E Regression (project-owned) runs the critical tier
           └── workflow_run fires compliance-evidence.yml's e2e job → uploads critical-tier evidence

day 2  operator reviews on the portal + clicks Approve UAT
       └── check-release-approval.yml's next eval flips green; PR is now mergeable

day 2  operator merges develop → main
       ├── post-deploy-prod.yml runs → smoke against prod, uploads production evidence
       └── Release flips to prod_review

day 2  operator clicks Approve Production on the portal
       └── Release flips to released → close-out-release.yml reconciles the ticket
```

**On a housekeeping release** (bare-date version, no REQ tag), the same five workflows run but with different content: no Tier 3 per-REQ artefacts, CI auto-generates the release ticket + security summary as a chore PR for operator sign-off, the four-eyes UAT step still applies to the bare-date release record. See [`change-workflows.md`](./change-workflows.md) for the per-shape detail.

## What you do if a gate is red

- **`Quality Gates` red.** Read the failing step's log. Most often a real failure — fix the code and push again. If it's a known flake, file an issue; the gate is binary and the framework rejects bypasses (`--no-verify` is a memorised anti-pattern). Document any cancel-and-admin-merge under the conditions in [`sdlc/files/_common/skills/sdlc-implementer/SKILL.md`](../sdlc/files/_common/skills/sdlc-implementer/SKILL.md) § _When an external gate hangs or fails for unrelated reasons_.
- **`Upload Compliance Evidence` red.** Often a network blip — re-trigger the workflow via _Actions → Re-run failed jobs_. If persistent, check `DEVAUDIT_API_KEY` is still issued (rotate via the portal at `/projects/<slug>/api-keys`) and `DEVAUDIT_BASE_URL` is current (the v0.1.54 redirect-follow + drift-warning in `upload-evidence.sh` will surface this loudly).
- **`Compliance Validation` red.** Read the script output — it names the missing artefact / unconventional commit. Authoring fix is local + a re-push.
- **`DevAudit Release Approval` red.** Expected before UAT approval. Open `/projects/<slug>/releases/<id>` and Approve (or Reject + change-request) on the portal.
- **`post-deploy-prod` red.** Production smoke failed. Don't panic — the release stays at `uat_approved`, the merge is already in; the question is whether to roll the merge back, hotfix forward, or accept-with-rationale. Document the decision on the release ticket; the portal's incident-report flow can capture it as evidence if it's substantive.
