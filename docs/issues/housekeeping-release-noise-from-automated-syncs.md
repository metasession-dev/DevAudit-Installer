# Framework: Suppress housekeeping release noise from automated syncs and bundle changes into the next REQ release

**Status:** Proposed / Ready for implementation
**Labels:** `enhancement`, `framework`, `ci`
**Priority:** High
**Milestone:** v0.1.71+

---

## Problem Statement

Every push to `develop` that touches files outside `paths-ignore` triggers the `CI Pipeline` workflow (`ci.yml`), which calls `derive-release-version.sh` to determine the release version. When the commit lacks a `[REQ-XXX]` tag (e.g. `chore:`, `docs:`, `build:` commits), the script falls through all five resolution steps and hits the **bare-date fallback** (`vYYYY.MM.DD`). The `register-release` job then calls `upload-evidence.sh` with `--create-release-if-missing`, creating a new housekeeping release record on the portal.

This means **automated tooling syncs** — `devaudit update`, Dependabot bumps, CI template regeneration — each create a separate housekeeping release on the portal, even though these are not operator-authored releases that warrant independent review and sign-off.

### Evidence (wawagardenbar-app)

| CI Run ID | Workflow | Event | Commit | Result |
|-----------|----------|-------|--------|--------|
| 28156639541 | CI Pipeline | push | `chore: sync DevAudit templates from v0.1.69 to v0.1.70` | Housekeeping release `v2026.06.25` created |
| 28150508465 | E2E Regression | schedule | (scheduled run on `main`) | Evidence uploaded to derived release |

The `devaudit update` sync commit has no `[REQ-XXX]` tag → bare-date fallback → `--create-release-if-missing` → portal release record. Each sync on a different day creates a new `vYYYY.MM.DD` release.

### Root Cause Chain

1. `devaudit update` pushes a `chore:` commit to `develop` (no REQ tag)
2. `ci.yml` triggers on push to `develop` (the commit touches files outside `paths-ignore`)
3. `derive-release-version.sh` steps 1–4 all fail (no REQ tag, no pending ticket, no RTM IN PROGRESS row)
4. Step 5 fallback: `v$(date -u +%Y.%m.%d)` → e.g. `v2026.06.25`
5. `register-release` job calls `upload-evidence.sh` with `--create-release-if-missing`
6. Portal creates a new release record for `v2026.06.25`
7. `compliance-evidence.yml` auto-generates stub release ticket + security summary, opens a sign-off PR

## Options Considered

### Option 1: `[skip ci]` on automated sync commits

Add `[skip ci]` to the commit message generated by `devaudit update` (and optionally Dependabot/Renovate). GitHub Actions natively skips workflows when the commit message contains `[skip ci]`.

**Pros:**
- Zero workflow template changes
- No portal-side changes
- Immediate effect — no release record created for sync commits

**Cons:**
- No gates run on the sync — a broken template wouldn't be caught until the next real push
- Housekeeping changes sit untested on `develop` until the next REQ-tagged commit triggers CI
- If there's a long gap between REQs, changes accumulate untested

### Option 2: Batch housekeeping into a periodic release

Accumulate chore/docs/dep changes and ship them on a weekly cadence. DevAudit's bare-date versioning already collapses same-day pushes into one `vYYYY.MM.DD` release.

**Pros:**
- Reduces portal noise to ~1 housekeeping release per week
- Gates still run on the batch

**Cons:**
- Requires operator discipline or a batching mechanism
- Still creates housekeeping releases (just fewer)
- Doesn't address the core issue: tooling syncs aren't operator-authored releases

### Option 3: Release on explicit trigger only (semantic-release pattern)

Only create release records for `feat`/`fix`/`refactor` commit types. Skip `chore`/`docs`/`ci`/`build` entirely in `register-release`.

**Pros:**
- Clean portal — only meaningful releases appear
- Aligns with industry standard (semantic-release, release-please)

**Cons:**
- Loses audit trail for dependency bumps and config changes
- ISO 27001 A.8.32 (change management) wants changes authorized, tested, documented — absorbing silently into the next REQ weakens the position
- Stricter frameworks (SOX, FDA) require explicit release records for production-affecting changes including dependency bumps

### Option 4: Selective skip + manual trigger (two-tier)

Low-risk syncs (`devaudit update`, docs, formatting) → `[skip ci]`. Dependency bumps and config changes → CI runs but no release created (commit-type filter in `register-release`).

**Pros:**
- Gates run on anything that could break the build
- Portal only shows operator-relevant releases

**Cons:**
- Still loses audit trail for dep bumps (they run through CI but don't get a release record)
- Complexity — two different skip mechanisms

### Option 5 (Selected): `[skip ci]` for tooling syncs + bundled changes summary on next REQ release

**Two-part solution:**

1. **`devaudit update` syncs** → `[skip ci]` in commit message → no CI, no release
2. **Next REQ-tagged commit** → CI runs (tests full `develop` state including skipped changes) → new `generate-bundled-changes.sh` script scans commits since the last release tag, identifies skipped housekeeping commits, generates a markdown summary → uploaded as `bundled_changes` evidence against the REQ release

**The portal shows for REQ-042:**
```
Release: REQ-042 — Add booking widget
├── Gate evidence: SAST ✓  Dep-audit ✓  E2E ✓  Build ✓
├── Release ticket: operator-authored (references bundled changes)
├── Bundled changes (auto-generated):
│   ├── chore: sync DevAudit templates v0.1.69 → v0.1.70
│   ├── chore: bump eslint 9.0.5 → 9.0.6
│   └── docs: update API reference for /bookings endpoint
└── Security summary: covers full develop state at CI time
```

**Pros:**
- No portal noise from automated syncs
- Complete audit trail — housekeeping changes are explicitly documented in the next REQ release
- Gate evidence covers the full code state (not just the REQ commit)
- Operator can review the bundled changes list before sign-off
- Aligns with SDLC process: the REQ release becomes the unit of release, housekeeping rides along

**Cons:**
- Housekeeping changes wait for the next REQ to be tested in CI (mitigated: operator can manually trigger CI via `workflow_dispatch` if needed)
- Requires new script + CI step
- If no REQ lands for a long time, housekeeping changes are untested (same as Option 1)

---

## Why Option 5 is the best fit for our SDLC process

### Alignment with the four-eyes lifecycle

Our SDLC defines two release shapes: **tracked** (`REQ-XXX`) and **housekeeping** (`vYYYY.MM.DD`). Both go through the same `draft → uat_review → uat_approved → prod_review → prod_approved → released` four-eyes lifecycle. The housekeeping path was designed for **operator-authored** ticketless work — a developer pushing `chore: bump eslint` or `docs: update API reference`.

Automated tooling syncs (`devaudit update`) are **not operator-authored changes**. They're machine-generated template refreshes. Creating a housekeeping release for each one pollutes the portal with release records that require human sign-off but carry no operator intent. This undermines the signal-to-noise ratio of the release queue.

### Audit trail preservation

The bundled changes summary ensures no change is invisible. When an auditor asks "what changed between REQ-042 and REQ-043?", the answer is on the portal — REQ-043's release record includes the explicit list of housekeeping commits it absorbed. This is a **stronger** audit position than the current state, where each housekeeping change creates a separate release that may or may not be properly signed off.

### Gate evidence coverage

CI gates (SAST, dep-audit, E2E, build) already test the **full state of `develop`** at the time they run. When REQ-042 triggers CI, the gates test the code including the skipped housekeeping changes. The gate evidence is attributed to REQ-042 and covers everything. No testing gap.

### Industry alignment

This approach mirrors how teams using semantic-release handle `chore`/`docs` commits — they don't create releases, but they're tested when the next `feat`/`fix` lands. The bundled changes summary adds the audit trail that semantic-release doesn't provide, making it **stronger** than the industry baseline.

### Minimal disruption

- No workflow template structural changes (no new conditionals in `register-release`)
- No portal API changes (uses existing evidence upload mechanism)
- One new script (`generate-bundled-changes.sh`) + one new CI step
- `[skip ci]` is a well-understood GitHub Actions convention

## Required Changes

### 1. Add `[skip ci]` to `devaudit update` commit message

**File:** `cli/src/update/commit-sync.ts` (or equivalent — the step that commits synced templates)

**Change:** Append `[skip ci]` to the commit message when `devaudit update` auto-commits synced files.

**Current (example):**
```
chore: sync DevAudit templates from v0.1.69 to v0.1.70
```

**After:**
```
chore: sync DevAudit templates from v0.1.69 to v0.1.70 [skip ci]
```

### 2. New script: `generate-bundled-changes.sh`

**File:** `sdlc/files/_common/scripts/generate-bundled-changes.sh`

**Purpose:** Scan commits since the last release tag (or last N commits if no tag), filter for housekeeping commit types (`chore`/`docs`/`ci`/`build`/`test`/`revert`), output a markdown summary.

**Usage:**
```bash
bash scripts/generate-bundled-changes.sh <since-ref> > compliance/pending-releases/BUNDLED-CHANGES-<version>.md
```

**Output format:**
```markdown
## Bundled Changes

The following housekeeping commits were absorbed into this release since `<since-ref>`:

- `abc1234` chore: sync DevAudit templates from v0.1.69 to v0.1.70
- `def5678` chore: bump eslint 9.0.5 → 9.0.6
- `ghi9012` docs: update API reference for /bookings endpoint

These commits were not individually released (tooling syncs / housekeeping).
Gate evidence above covers the full state of develop at CI time, including these changes.
```

### 3. New CI step in `ci.yml.template`: upload bundled changes

**File:** `sdlc/files/ci/ci.yml.template`

**Location:** `register-release` job, after "Ensure release exists" step.

**Logic:**
- Only runs when the derived version is a REQ (not bare-date)
- Calls `generate-bundled-changes.sh` with the last release tag as `since-ref`
- Uploads the output as `bundled_changes` evidence type against the REQ release

### 4. Update housekeeping release playbook

**File:** `docs/release-playbooks/housekeeping-release.md`

**Add:** Note that `devaudit update` syncs no longer create housekeeping releases (they use `[skip ci]`), and that housekeeping changes are bundled into the next REQ release with an auto-generated summary.

### 5. Update `derive-release-version.sh` documentation

**File:** `sdlc/files/_common/scripts/derive-release-version.sh`

**Add:** Comment noting that `[skip ci]` commits won't trigger CI at all, so the bare-date fallback is now only reached by human-authored housekeeping commits (the intended use case).

## Acceptance Criteria

- [ ] `devaudit update` auto-commit message includes `[skip ci]`
- [ ] `devaudit update` sync pushes do not trigger `ci.yml` or `compliance-evidence.yml`
- [ ] `devaudit update` sync pushes do not create a portal release record
- [ ] `generate-bundled-changes.sh` script exists and correctly filters housekeeping commit types
- [ ] `ci.yml` `register-release` job includes a step to generate + upload bundled changes for REQ-tagged releases
- [ ] Bundled changes evidence appears on the portal under the REQ release
- [ ] Human-authored `chore`/`docs` commits (without `[skip ci]`) still create housekeeping releases as before
- [ ] Gate evidence on REQ releases covers the full `develop` state including skipped housekeeping changes
- [ ] Housekeeping release playbook updated to document the new behaviour
- [ ] Test: a `devaudit update` sync followed by a REQ-tagged commit results in one REQ release with bundled changes summary

---

## Out of Scope

- Portal-side rendering of `bundled_changes` evidence type (portal repo, separate change)
- Applying `[skip ci]` to Dependabot/Renovate commits (consumers can configure this in their own `dependabot.yml`)
- Changing the housekeeping release path for human-authored commits (still works as designed)
- Batch release scheduling (could be a future enhancement if portal noise remains an issue)

---

## Dependencies

- None blocking. The `[skip ci]` change is self-contained in the CLI.
- The `generate-bundled-changes.sh` script + CI step are additive (new evidence type).
- Portal may need to recognise `bundled_changes` evidence type for nice rendering, but the upload mechanism is the same as any other evidence.
